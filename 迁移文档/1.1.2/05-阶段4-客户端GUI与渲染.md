## 0. 设计原则：从优生成，非原代码迁移

原项目（RTSbuilding-main 1.21.1）的客户端架构存在严重的低效问题。本阶段选择**从优设计新架构**，而非盲目搬运原始代码。原因如下：

### 0.1 原项目低效问题诊断

| 问题 | 严重程度 | 表现 |
|------|---------|------|
| **God Object 反模式** | 🔴 严重 | `BuilderScreen.java` 2426 行、`ClientRtsController.java` 3155 行 —— 两个类承担了渲染、输入路由、状态管理、网络分发、面板调度、相机控制等所有职责 |
| **无分层架构** | 🔴 严重 | 渲染 `GuiGraphics.fill()` 调用和网络 `PacketDistributor.sendToServer()` 混在同一行代码块中，无逻辑/视图分离 |
| **状态散落** | 🟡 中等 | 29+ 个散落字段（`hoveredEntry`, `hoveredRecentEntry`, `hoveredFluidEntry`, `hoveredToolSlot`...）分布在多个类中，无统一状态管理 |
| **输入路由混乱** | 🟡 中等 | `mouseClicked()`, `keyPressed()` 方法通过 10+ 层 if-else 瀑布式路由到不同面板，每新增一个面板需手动插桩 |
| **渲染层过重** | 🟡 中等 | 依赖 NeoForge `GuiGraphics`/`PoseStack`/`MultiBufferSource` 等高级 API，到 1.7.10 需要逐行降级到 `GL11` + `Tessellator` |
| **UI 缩放复杂度** | 🟡 中等 | `fixedRtsScaleRenderPass` + `fixedRtsScaleInputPass` 双标志位 + width/height 临时篡改实现 UI 缩放，侵入每个鼠标/渲染路径 |
| **ClientRtsController 超载** | 🔴 严重 | 同时管理：相机状态(~20字段)、存储页面(~10字段)、合成面板(~10字段)、进度数据(~10字段)、任务检测(~10字段)、绑定管理(~10字段)、输入灵敏度、模式切换、采矿状态……相当于把所有 ViewModel 塞进一个类 |
| **BottomPanel 巨型渲染** | 🟡 中等 | `BottomPanel.java` 2015 行，9 种标签页渲染（存储/分类/合成/流体/蓝图/创意/快速建造/设置/指南），通过 if-else 分支 |
| **原 RtsStorageManager** | 🔴 严重 | 2416 行，服务端已拆分为多个子模块（阶段 3 成果），但客户端存储状态全部堆积在 ClientRtsController 中 |

### 0.2 优化后的架构目标

**核心优化**：
1. **单层容器** — 一个 `RtsScreen extends GuiScreen` 替代原 `BuilderScreen` 2426 行巨兽
2. **输入路由独立** — `RtsInputRouter` 统一处理鼠标/键盘分发，面板只需声明自己的交互区域
3. **状态集中管理** — `RtsClientState` 作为单一 ViewModel，替代散落在 Controller 中的 50+ 字段
4. **面板协议化** — `IRtsPanel` 接口定义 `render() / hitTest() / onClick() / onScroll()`，所有面板实现它
5. **世界渲染模块化** — `RtsWorldRenderer` 拆分为 4 个独立渲染器，每个 ≤150 行
6. **1.7.10 原生渲染** — 直接使用 `GL11` + `Tessellator` + `Minecraft.fontRenderer`，不包装抽象层

## 2. 优化架构详细设计

### 2.1 RtsScreen — 主容器（替代 BuilderScreen）

```java
public class RtsScreen extends GuiScreen {
    // —— 核心依赖 ——
    private final RtsClientState state;           // 集中式状态
    private final RtsInputRouter inputRouter;     // 输入路由
    private final RtsWorldRenderer worldRenderer; // 世界叠加层

    // —— 面板（实现 IRtsPanel） ——
    private final Map<String, IRtsPanel> panels = new LinkedHashMap<>();

    @Override
    public void initGui() {
        // 1. 创建/恢复状态
        // 2. 初始化所有面板
        // 3. 注册面板到输入路由器
        // 4. 请求初始服务端数据（存储页、合成列表、进度状态）
    }

    @Override
    public void drawScreen(int mouseX, int mouseY, float partialTicks) {
        // 1. 重置帧级 hover 状态
        // 2. 渲染背景 + 所有面板（按 Z 序）
        // 3. 渲染叠加层（伤害闪烁、任务检测、存储扫描）
        // 4. 渲染世界交互光标
        // 5. 渲染 tooltip
    }

    @Override
    protected void mouseClicked(int mouseX, int mouseY, int button) {
        // 委托给 inputRouter
        inputRouter.dispatchClick(mouseX, mouseY, button);
    }

    @Override
    protected void keyTyped(char c, int keyCode) {
        // 委托给 inputRouter
        inputRouter.dispatchKey(c, keyCode);
    }

    @Override
    public void onGuiClosed() {
        // 持久化状态 + 清理资源
        state.persist();
        worldRenderer.cleanup();
    }
}
```

**关键简化**：
- 去掉原 BuilderScreen 的 `fixedRtsScaleRenderPass` / `fixedRtsScaleInputPass` 双通道缩放机制 —— 1.7.10 的 GuiScreen 天然支持 `ScaledResolution`
- 去掉所有 if-else 瀑布式分发 —— 由 `RtsInputRouter` 的 hit-test 链式调用替代
- 去掉直接对 panel 子类的字段引用 —— 通过 `IRtsPanel` 接口/注册表通信

### 2.2 IRtsPanel — 面板协议

```java
public interface IRtsPanel {
    /** 面板名称（用于识别和 Z-序排序） */
    String panelName();

    /** 面板在当前屏幕上的矩形边界（用于 hit-test） */
    Rectangle getBounds();

    /** 渲染面板内容 */
    void render(GuiScreen screen, int mouseX, int mouseY, float partialTicks);

    /** 鼠标点击 hit-test：返回 true 表示消费了点击 */
    boolean onMouseClick(int mouseX, int mouseY, int button);

    /** 鼠标滚轮 */
    boolean onMouseScroll(int mouseX, int mouseY, int scroll);

    /** 键盘输入（仅在面板"聚焦"时调用） */
    boolean onKeyTyped(char c, int keyCode);

    /** 面板是否可见/可交互 */
    boolean isVisible();

    /** 帧级重置（在每帧 render 开始时调用） */
    void resetFrameState();
}
```

**优势**：
- 新增面板只需实现协议 + 注册，不修改 RtsScreen
- 面板之间完全解耦，通过 RtsClientState 共享数据
- hit-test 统一在输入路由器中以 Z-序处理

### 2.3 RtsInputRouter — 输入分发

```java
public class RtsInputRouter {
    private final List<IRtsPanel> panels; // 按 Z-序排列（后渲染的 = 上层）

    public boolean dispatchClick(int mouseX, int mouseY, int button) {
        // 从上层到下层遍历面板
        for (int i = panels.size() - 1; i >= 0; i--) {
            IRtsPanel panel = panels.get(i);
            if (panel.isVisible() && panel.getBounds().contains(mouseX, mouseY)) {
                if (panel.onMouseClick(mouseX, mouseY, button)) {
                    return true; // 消费
                }
            }
        }
        // 面板未消费 → 传递给世界交互处理
        return dispatchWorldClick(mouseX, mouseY, button);
    }
}
```

**替代原 BuilderScreen 的 10+ 层 if-else 瀑布**。所有面板的交互优先级由 Z-序自动决定，无需手动编入 if-else 链。

### 2.6 RtsWorldRenderer — 世界叠加层渲染（替代 RtsVisualOverlayRenderer）

基于 1.7.10 `ForgeHooksClient` 的 `RenderWorldLastEvent`：

```java
@SubscribeEvent
public void onRenderWorldLast(RenderWorldLastEvent event) {
    if (!RtsClientState.get().camera.hasBounds()) return;

    Minecraft mc = Minecraft.getMinecraft();
    EntityPlayer player = mc.thePlayer;
    double px = player.lastTickPosX + (player.posX - player.lastTickPosX) * event.partialTicks;
    double py = player.lastTickPosY + (player.posY - player.lastTickPosY) * event.partialTicks;
    double pz = player.lastTickPosZ + (player.posZ - player.lastTickPosZ) * event.partialTicks;

    GL11.glPushMatrix();
    GL11.glTranslated(-px, -py, -pz);

    boundaryRenderer.render(...);
    storageHighlightRenderer.render(...);
    interactionTargetRenderer.render(...);
    shapeGhostRenderer.render(...);
    animationRenderer.render(...);

    GL11.glPopMatrix();
}
```

#### 2.6.1 各子渲染器 1.7.10 实现方案

| 渲染器 | 1.7.10 API | 说明 |
|--------|-----------|------|
| `BoundaryRenderer` | `GL11.glBegin(GL_LINES)` + `Tessellator` | 红色半透明边界线框 |
| `StorageHighlightRenderer` | `GL11.glBegin(GL_QUADS)` + `Tessellator` | 已链接存储方块半透明高亮 |
| `InteractionTargetRenderer` | `Tessellator` + `GL11.glPolygonOffset` | 鼠标悬停目标线框（使用原 BRACKET_QUADS 技术） |
| `ShapeGhostRenderer` | `Tessellator` + 半透明方块 | 形状建造预览（Placement Ghost） |
| `AnimationRenderer` | `Tessellator` + 粒子效果 | 放置/破坏动画 |

**关键降级**：
- **PoseStack → GL11.glPushMatrix/glPopMatrix/glTranslate/glRotate**
- **MultiBufferSource → Tessellator（单例）+ BufferBuilder**
- **RenderSystem → GL11 直接调用 + GLStateManager**
- **RenderType → 手动管理 GL 状态**

## 6. 风险与注意事项

| 风险 | 缓解措施 |
|------|---------|
| **1.7.10 `Tessellator` API 差异** | Forge 1.7.10 有扩展版 `Tessellator`（通过 MCP），API 与 1.21 `BufferBuilder` 不同但功能相同。优先使用稳定子集 |
| **RenderItem 渲染物品** | `RenderItem.renderItemAndEffectIntoGUI()` 的签名在 1.7.10 需要 `FontRenderer` + `TextureManager` 参数 |
| **UI 缩放** | 1.7.10 的 `ScaledResolution` 自身有 GUI 缩放支持。RTS 额外缩放可通过临时调整 `width`/`height` 或使用 `GL11.glScalef` 实现 |
| **RtsCameraEntityRenderer** | 需要手写 `Render` 子类，渲染一个简单的线框/图标表示相机位置 |
| **蓝图 GUI 复杂度** | 原 BlueprintPanel 相关代码约 800 行，结构尚可。基本可直接复用适配到 1.7.10 IRtsPanel 接口 |
| **拼音搜索** | `RtsPinyinSearch.java` 阶段 3 已完成迁移，客户端直接调用 |
