第二阶段建立了工作流引擎和管线系统的骨架，但有若干 no-op 占位和未统一的入口。本阶段目标：

1. 实现 4 个工作流网络消息，激活客户端-服务端双向同步
2. 统一挖掘/放置网络入口走管线系统
3. 激活 `RtsWorkflowSyncService`、`PlacementExecutePipe` 等预留实现
4. 补充客户端工作流状态接收和 UI 反馈

## 三、工作流网络消息实现（4 个）

### 3.1 消息文件清单

| # | 文件路径 | 方向 | 说明 |
|---|---------|------|------|
| 1 | `network/builder/C2SRtsDeleteWorkflowMessage.java` | C→S | 删除指定工作流 |
| 2 | `network/builder/C2SRtsPauseWorkflowMessage.java` | C→S | 暂停/恢复指定工作流 |
| 3 | `network/builder/S2CRtsWorkflowProgressMessage.java` | S→C | 单个工作流进度推送 |
| 4 | `network/builder/S2CRtsWorkflowProgressBatchMessage.java` | S→C | 批量工作流进度推送 |

### 3.2 实现要点

**C2S 消息（客户端→服务端）**:
- 使用 1.7.10 `IMessage` + `IMessageHandler` 模式（对齐现有 `C2SRtsMineMessage` 等）
- `C2SRtsDeleteWorkflowMessage` — 携带 `int workflowEntryId`，Handler 调用 `RtsWorkflowEngine.getInstance().deleteWorkflow(player, entryId)`
- `C2SRtsPauseWorkflowMessage` — 携带 `int entryId`（-1 表示切换全部），Handler 调用 `token.pause()` / `token.unpause()`

**S2C 消息（服务端→客户端）**:
- 使用 1.7.10 `IMessage` + `IMessageHandler` 模式
- `S2CRtsWorkflowProgressMessage` — 编码工作流快照的所有字段（类型、优先级、进度、状态、缺失物品等）
- `S2CRtsWorkflowProgressBatchMessage` — 批量编码多条进度消息，减少网络开销
- 线格式对齐 main 版本的 wire format，但用 `ByteBuf` 替代 `RegistryFriendlyByteBuf`

**注册**:
- 在 `RtsNetworkManager.registerMessages()` 的 Builder 区域末尾追加 4 个 `disc++` 注册

### 3.3 1.7.10 适配要点

| main 版本实现 | 1.7.10 适配方案 |
|--------------|----------------|
| `record` 类型 | final class + 手动 getter + 构造器 |
| `RegistryFriendlyByteBuf` | `ByteBuf` / `ByteArrayDataInput` / `ByteArrayDataOutput` |
| `StreamCodec` | `IMessage.fromBytes` / `IMessage.toBytes` |
| `CustomPacketPayload` | `IMessage` + `IMessageHandler` |
| `List.of()` | `Collections.emptyList()` |

## 四、激活 RtsWorkflowSyncService

### 4.1 修改文件

| 文件 | 修改内容 |
|------|----------|
| `server/workflow/service/RtsWorkflowSyncService.java` | 移除 no-op，实现真实 packet 发送 |
| `client/WorkflowViewModel.java` | **新建** — 客户端接收并缓存工作流状态 |

### 4.2 实现要点

**服务端发送逻辑** (`RtsWorkflowSyncService.notifyPlayer`):
1. 从 `RtsWorkflowSlotManager` 获取所有 occupiedEntries
2. 构建 `S2CRtsWorkflowProgressBatchPayload`
3. 通过 `RtsNetworkManager.NETWORK.sendTo(payload, player)` 发送

**客户端接收逻辑** (`WorkflowViewModel`):
1. 静态持有 `List<RtsWorkflowStatus>` 缓存
2. `S2CRtsWorkflowProgressMessage.Handler` 收到消息后更新缓存
3. 提供 `getProgress(int entryId)` / `getAllProgress()` 查询接口
4. 供客户端 UI 渲染工作流状态行

**触发时机**:
- `RtsWorkflowEngine.start()` → `notifyPlayer()`
- `RtsWorkflowToken.markProgress()` → `notifyPlayer()`
- `RtsWorkflowToken.complete()` → `notifyPlayer()` (idle)
- `RtsWorkflowEngine.cleanupStaleWorkflows()` → `notifyPlayer()`

### 5.2 实现要点

**改造模式**（对齐 `C2SRtsMineMessage` → `MINE_SINGLE` 管线的已有模式）:

```
// C2SRtsUltimineMessage.Handler.onMessage() 伪代码
Map<String, Object> args = new HashMap<>();
args.put(UltimineExecutePipe.KEY_X.name(), msg.posX);
args.put(UltimineExecutePipe.KEY_Y.name(), msg.posY);
args.put(UltimineExecutePipe.KEY_Z.name(), msg.posZ);
args.put(MiningExecutePipe.KEY_FACE.name(), msg.face);
args.put(MiningExecutePipe.KEY_TOOL_SLOT.name(), msg.toolSlot);
args.put(MiningExecutePipe.KEY_TOOL_ITEM_ID.name(), msg.toolItemId);
args.put(MiningExecutePipe.KEY_TOOL_PROTOTYPE.name(), toolPrototype);
args.put(UltimineExecutePipe.KEY_LIMIT.name(), msg.limit);
args.put(UltimineExecutePipe.KEY_MODE.name(), msg.mode);
WorkflowPipeline<PipelineContext> pipeline = PipelineRegistry.get(RtsWorkflowType.ULTIMINE);
```

**防递归**:
- `RtsMineManager.startUltimine()` 改为先走管线，失败时回退到 `startUltimineDirect()`
- `UltimineExecutePipe` 调用 `RtsMineManager.startUltimineDirect()`

**进度验证**:
- 管线入口保留原有的 `RtsProgressionManager.canUse()` 检查
- 由 `ProgressionGatePipe` 统一执行，消息 Handler 中不再重复检查

## 十二、Java 版本说明

GTNH 通过 JVM Downgrader 和 UniMixins 在 Java 25 上运行 1.7.10 Forge。编译目标仍然是 J8 bytecode（通过 Jabel），但开发时可以使用更高版本的 Java 语法特性。本阶段实现中：
- 仍优先使用 Java 8 兼容写法以保持一致性
- 若 Jabel 支持 `record` 等特性，可评估是否逐步采用
- 网络消息仍使用 `IMessage` / `IMessageHandler` 模式（1.7.10 Forge 标准）
