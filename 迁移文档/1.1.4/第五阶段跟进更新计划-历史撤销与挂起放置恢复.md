## 二、剩余 stub Handler 现状

### 2.1 三个完全空实现的 Handler

| 消息 | 位置 | 方向 | 字段 | 当前 Handler |
|------|------|------|------|------------|
| `C2SRtsQuickDropMessage` | `network/builder/` | C→S | itemId, amount, dropX/Y/Z | `return null` |
| `C2SRtsPlaceFluidMessage` | `network/builder/` | C→S | clickedX/Y/Z, face, hitX/Y/Z, forcePlace, fluidId, ray 方向/原点 | `return null` |
| `C2SRtsSetModeMessage` | `network/builder/` | C→S | mode (byte) | `return null` |

### 2.2 预期行为

**C2SRtsQuickDropMessage** — 快捷丢弃物品：
1. 服务端收到后根据 `itemId` 从链接存储提取对应物品
2. 在 `(dropX, dropY, dropZ)` 位置生成 `EntityItem` 实体
3. 刷新存储页面

**C2SRtsPlaceFluidMessage** — 远程放置流体：
1. 服务端根据 `fluidId` 创建对应 Fluid 方块
2. 使用 `face` 确定放置面，`(hitX, hitY, hitZ)` 为点击位置
3. 调用 `RtsStorageManager` 消耗流体并放置
4. `forcePlace` 强制放置标志（跳过碰撞检查）

**C2SRtsSetModeMessage** — 设置构建模式：
1. 服务端收到后在 RTS 会话中设置 `BuilderMode`
2. 同步更新到 `RtsClientState` 缓存
3. 可能需要返回确认消息给客户端

## 三、历史/撤销系统

### 3.1 背景分析

语言文件（`en_US.lang` 等）已预置撤销/重做的 UI 字符串和快捷键提示：
- 撤销: `Ctrl+Z`
- 重做: `Ctrl+Y`

表明 UI 层已有预期，但服务端实现完全空缺。

### 3.2 缺失文件清单（6 个）

```
src/main/java/com/rtsbuilding/rtsbuilding/
├── server/history/
│   ├── ServerHistoryManager.java          — 历史管理器（全局单例）
│   ├── HistoryExecutor.java               — 撤销/重做执行器
│   ├── HistoryEntry.java                  — 历史条目（NBT 序列化）
│   └── HistoryBlockRecord.java            — 单方块记录（before/after 状态）
└── network/builder/
    ├── C2SRtsUndoMessage.java             — 撤销请求 C→S
    └── S2CRtsHistorySyncMessage.java      — 历史同步 S→C
```

### 3.3 数据模型设计

**HistoryBlockRecord** — 单方块变更记录：
```
fields:
  - int x, y, z            — 方块位置
  - Block oldBlock         — 变更前方块（可为 null）
  - int oldMeta            — 变更前 meta
  - Block newBlock         — 变更后方块（可为 null）
  - int newMeta            — 变更后 meta
  - NBTTagCompound oldTile — 变更前 TileEntity NBT（可为 null）
  - NBTTagCompound newTile — 变更后方块 TileEntity NBT（可为 null）
```

**HistoryEntry** — 单次操作历史条目：
```
fields:
  - long timestamp          — 操作时间戳
  - RtsWorkflowType type    — 操作类型（PLACE_SINGLE / MINE_SINGLE / ULTIMINE / AREA_MINE / AREA_DESTROY）
  - List<HistoryBlockRecord> records — 本次操作涉及的所有方块变更
```

**ServerHistoryManager** — 全局单例管理器：
```
职责:
  - Map<UUID, Deque<HistoryEntry>> perPlayerHistory (每玩家双端队列，最多 100 条)
  - Map<UUID, Deque<HistoryEntry>> perPlayerRedoStack (重做栈)
  - record(player, HistoryEntry)  — 追加历史记录
  - undo(player)                  — 撤销最近一次操作（反向执行所有 BlockRecord）
  - redo(player)                  — 重做最近一次撤销
  - clear(player)                 — 清空历史
  - 持久化: 写入 <world>/rtsbuilding/history_data.dat（可选，本阶段暂不实现持久化）
```

**HistoryExecutor** — 撤销/重做执行器：
```
核心逻辑:
  undo(player):
    取 perPlayerHistory 末尾 entry
    逆序遍历 entry.records:
      如果 record.oldBlock 为 null → 破坏 newBlock（setBlockToAir）
      否则 → 恢复 oldBlock + oldMeta + oldTile（setBlock + restore NBT）
    将 entry 推入 perPlayerRedoStack

  redo(player):
    取 perPlayerRedoStack 末尾 entry
    正序遍历 entry.records:
      如果 record.newBlock 为 null → 破坏 oldBlock
      否则 → 放置 newBlock + newMeta + newTile
    将 entry 推回 perPlayerHistory
```

### 3.4 网络消息设计

**C2SRtsUndoMessage** — 客户端撤销请求：
```
fields:
  - boolean redo     — true=重做, false=撤销
  - int count        — 批量撤销数量（当前版本固定为 1）

Handler:
  获取 player → ServerHistoryManager.getInstance().undo(player) / redo(player)
  → 返回 S2CRtsHistorySyncMessage(cleared=true/false, count=n)
```

**S2CRtsHistorySyncMessage** — 同步历史状态到客户端：
```
fields:
  - boolean cleared  — 是否清空
  - int count        — 当前历史条目数量

Handler（客户端）:
  更新 PlacementHistoryManager / RtsClientState
  刷新 UI（撤销按钮可用性）
```

### 3.5 管线集成

**HistoryRecordPipe** — 在放置/挖掘管线末尾记录历史：
- 当前 `HistoryRecordPipe` 已经存在（`server/pipeline/sync/`）但为 no-op 桩
- 本阶段实现完整记录逻辑：
  1. 从 PipelineContext 获取操作类型（KEY_WORKFLOW_TYPE）
  2. 从上下文获取操作涉及的方块列表（新增 `KEY_BLOCK_RECORDS` TypedKey）
  3. 构造 `HistoryEntry` 并调用 `ServerHistoryManager.record()`

**管线注册更新** — 在放置/挖掘管线末尾（UiRefreshPipe 之后）追加 HistoryRecordPipe：
```
MINE_SINGLE:  ... → ToolReturnPipe → WorkflowProgressPipe → NetworkSyncPipe → UiRefreshPipe → HistoryRecordPipe
ULTIMINE:     ... → ToolReturnPipe → WorkflowProgressPipe → NetworkSyncPipe → UiRefreshPipe → HistoryRecordPipe
AREA_MINE:    ... → ToolReturnPipe → WorkflowProgressPipe → NetworkSyncPipe → UiRefreshPipe → HistoryRecordPipe
AREA_DESTROY: ... → ToolReturnPipe → WorkflowProgressPipe → NetworkSyncPipe → UiRefreshPipe → HistoryRecordPipe
PLACE_SINGLE: ... → PlacementExecutePipe → UiRefreshPipe → HistoryRecordPipe
PLACE_BATCH:  ... → PlacementExecutePipe → PendingPlacementPipe → UiRefreshPipe → HistoryRecordPipe
QUICK_BUILD:  ... → PlacementExecutePipe → UiRefreshPipe → HistoryRecordPipe
```

**PipelineContext 扩展** — 新增 TypedKey：
```
PipelineContext 新增:
  TypedKey<List<HistoryBlockRecord>> KEY_BLOCK_RECORDS
```

> **注意**: 当前 PipelineContext 以 `Map<String, Object>` 存储数据。`TypedKey` 已有 `name()` 方法返回字符串 key。KEY_BLOCK_RECORDS 定义为 `TypedKey<List<HistoryBlockRecord>> KEY_BLOCK_RECORDS = new TypedKey<>("block_records")`。

## 四、挂起放置恢复系统

### 4.1 背景分析

`PendingPlacementPipe` 当前在 PLACE_BATCH 管线中作为 no-op 桩存在（`return PipelineResult.success()`）。main 版本中挂起放置是指：

- 批量放置时，某些方块因供需不足、区域限制等原因暂时无法放置
- 这些"挂起"的放置请求被保存
- 玩家后续可以恢复/取消这些挂起放置

### 4.2 缺失文件清单（7 个）

```
src/main/java/com/rtsbuilding/rtsbuilding/
├── server/service/
│   ├── RtsPendingPlacementService.java    — 挂起放置管理服务
│   └── RtsPlacedRecoveryService.java      — 放置恢复扫描服务
├── network/builder/
│   ├── C2SRtsSubmitPendingMessage.java    — 提交待处理 C→S
│   ├── C2SRtsScanResumePlacementMessage.java — 扫描可恢复 C→S
│   ├── C2SRtsResumePlacementActionMessage.java — 执行恢复 C→S
│   └── S2CRtsResumePlacementScanMessage.java — 恢复扫描结果 S→C
└── client/panel/
    └── RtsResumePlacementPanel.java       — 恢复放置面板（本阶段暂不实现）
```

### 4.3 服务设计

**RtsPendingPlacementService** — 挂起放置管理器：
```
字段:
  - Map<UUID, List<PendingBlock>> playerPendings  — 每玩家挂起列表

PendingBlock:
  - int x, y, z            — 目标位置
  - String blockId         — 方块 ID
  - int meta               — 方块 meta
  - byte face              — 放置面
  - long timestamp         — 挂起时间

方法:
  - addPending(player, itemId, meta, x, y, z, face)  — 添加挂起
  - List<PendingBlock> getPendings(player)            — 获取挂起列表
  - removePending(player, index)                      — 移除单个挂起
  - clearPendings(player)                             — 清空所有挂起
  - boolean hasPendings(player)                       — 是否有挂起
```

**RtsPlacedRecoveryService** — 放置恢复扫描服务：
```
职责:
  - 扫描服务器存盘数据，找出因崩溃/掉线而丢失的放置草稿
  - 返回可恢复的放置列表

方法:
  - List<RecoveryBlock> scanRecoverable(player)
    遍历存储管理器中的草稿数据，判断哪些可以恢复
```

### 4.4 网络消息设计

**C2SRtsSubmitPendingMessage** — 客户端提交待处理操作：
```
fields:
  - 无（服务端根据当前上下文处理）

Handler:
  获取 player → RtsPendingPlacementService 执行所有挂起放置
  → 返回成功/失败结果
```

**C2SRtsScanResumePlacementMessage** — 客户端请求扫描可恢复放置：
```
fields:
  - 无

Handler:
  获取 player → RtsPlacedRecoveryService.scanRecoverable(player)
  → 返回 S2CRtsResumePlacementScanMessage
```

**C2SRtsResumePlacementActionMessage** — 客户端请求执行某个恢复操作：
```
fields:
  - int actionIndex     — 恢复动作索引（对应扫描结果的第几项）
  - boolean executeAll  — 是否执行全部

Handler:
  根据 actionIndex 或 executeAll 执行对应恢复
```

**S2CRtsResumePlacementScanMessage** — 服务端返回扫描结果：
```
fields:
  - int count                         — 可恢复项数量
  - int posX[], posY[], posZ[]       — 位置数组
  - String blockId[]                  — 方块 ID 数组
  - int meta[]                        — meta 数组

Handler（客户端）:
  存储到 RtsClientState
  如果 count > 0，显示恢复提示/高亮
```

### 4.5 PendingPlacementPipe 实现

当前 no-op 桩替换为真实逻辑：

```java
@Override
public PipelineResult execute(PipelineContext ctx) {
    // 1. 检查放置结果：从上下文中取 KEY_PLACE_RESULTS
    // 2. 筛选失败的方块（供需不足等原因）
    // 3. 将失败方块添加到 RtsPendingPlacementService
    // 4. 如果有挂起项，返回 success 并设置 KEY_HAS_PENDING = true
    //    客户端据此显示"有挂起放置可恢复"提示
    return PipelineResult.success();
}
```

### 4.6 管线上下文扩展

```
PipelineContext 新增:
  TypedKey<List<PlaceResult>> KEY_PLACE_RESULTS     — 放置执行结果列表
  TypedKey<Boolean>          KEY_HAS_PENDING        — 是否有挂起放置
```

> **注意**: `PlaceResult` 在 PlacementExecutePipe 相关逻辑中已存在对应概念（放置结果的成功/失败/跳过枚举）。如果当前不存在此类型，需新建 `PlaceResult` 类（SUCCESS/FAILED/SKIPPED + 原因字段）。
