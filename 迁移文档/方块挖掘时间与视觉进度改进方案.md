# RTSbuilding 1.7.10 — 方块挖掘时间与视觉进度改进方案

## 一、现状问题分析

### 1.1 破坏时间过短（核心问题）

**文件**: `RtsMineManager.java` `:149`

当前速度计算公式：

```java
speedMultiplier = Math.min(toolSpeed / Math.max(hardness, 0.3F), 5.0F);
// 每 tick 进度 = BASE_PROGRESS_PER_TICK(1/30) × speedMultiplier
```

**对比原版 Minecraft 1.7.10 的公式** (`ForgeHooks.blockStrength()`):

```java
// 能采集: breakSpeed / hardness / 30
// 不能采集: breakSpeed / hardness / 100
```

#### 差异点：

| 因素 | 移植版 | 原版 MC 1.7.10 | 原版 RTSbuilding 1.21.1 |
|------|--------|----------------|------------------------|
| 基础公式 | `1/30 × toolSpeed / max(hardness, 0.3)` | `breakSpeed / hardness / 30` | `getDestroyProgress()` (调用原版) |
| 硬度最小值 | clamp 到 `0.3F` | 无 clamp，直接用原值 | 无 clamp |
| 不能采集的方块 | **与能采集的速度相同** | 慢 3.33 倍 (`/100` 而非 `/30`) | 原版 API 自动处理 |
| 水下挖掘 | 无处理 | 速度 /5 | 取消惩罚（远程挖掘不在水下） |
| 不在地面 | 无处理 | 速度 /5 | 临时设 `onGround=true` |
| 效率附魔 (Efficiency) | 仅用 `getDigSpeed()` 基础值 | `getDigSpeed()` + `i²+1` 额外加成 | 原版 API 自动包含 |
| 急迫药水 (Haste) | 无处理 | `× (1 + 0.2×(amplifier+1))` | 原版 API 自动包含 |
| 挖掘疲劳 (Mining Fatigue) | 无处理 | `× (1 - 0.2×(amplifier+1))` | 原版 API 自动包含 |
| 首次点击延迟 | 无 | 5 tick (0.25秒) | 无（远程挖掘） |

#### 数值对比示例（铁镐挖石头，hardness=1.5）：

| 实现 | speedMultiplier / 每tick进度 | 总破坏时间 |
|------|-------------------------------|-----------|
| **移植版** (当前) | `6.0/1.5=4.0` → 每tick `0.0333×4=0.133` | **~8 tick (0.4秒)** |
| **原版 MC 1.7.10** | `6.0/1.5/30=0.133` 每tick | **~8 tick (0.4秒)** |
| **原版 MC 1.7.10** (空手不能采集) | `1.0/1.5/100=0.0067` 每tick | **~150 tick (7.5秒)** |
| **移植版** (空手) | `1.0/1.5=0.667` → 每tick `0.0333×0.667=0.022` | **~45 tick (2.25秒)** |

**关键发现**：铁镐挖石头的速度两者基本一致（~8 tick），但 **空手挖石头** 差异巨大：
- 移植版：2.25 秒（应该很慢但不够慢）
- 原版 MC：7.5 秒（不能采集时慢 3.33 倍）

**真正导致"过快"的原因**是 `hardness` 的 clamp：
- 移植版把 `max(hardness, 0.3)` 使得硬度为 0 的方块（如草方块）也至少需要 0.3 的硬度
- 但硬度为 0.3 的方块（如沙砾、沙子）本应很快破坏，却被提高了硬度下限

**更关键的问题**：移植版完全忽略了"不能采集时速度除以 100"的规则。这导致：
- 空手挖石头：移植版 2.25 秒 vs 原版 7.5 秒
- 木镐挖铁矿（不能采集）：移植版 ~1 秒 vs 原版 ~10 秒

---

### 1.2 没有原版方块破坏进度纹理

**文件**: `AnimationRenderer.java` `:68-136`

当前渲染方式：自定义 `GL_LINES` 红色半透明网格覆盖层。

原版方式：`World.destroyBlockInWorldPartially(entityId, x, y, z, stage)` → `RenderGlobal.drawBlockDamageTexture()` → 标准 `destroy_stage_0~9.png` 裂纹纹理。

**差异**：
- 移植版的红色网格是自定义渲染，视觉效果与原版完全不同
- 原版 RTSbuilding 1.21.1 使用 `level.destroyBlockProgress(RTS_MINE_RENDER_ID, pos, stage)` 委托给原版裂纹系统
- 1.7.10 中对应的方法是 `World.destroyBlockInWorldPartially(entityId, x, y, z, stage)`

---

### 1.3 缺少首次点击延迟

原版 MC 1.7.10 的 `PlayerControllerMP` 有 `blockHitDelay = 5`（首次点击后 5 tick 延迟），模拟玩家点击-等待-持续挖掘的行为。移植版没有此延迟。

## 二、改进方案

### 2.1 修复破坏速度计算（对齐原版 MC 1.7.10）

**目标文件**: `RtsMineManager.java`

**方案**: 使用 Forge 的 `ForgeHooks.blockStrength()` 或手动实现等价逻辑。

#### 方案 A：直接调用 Forge API（推荐）

```java
// 在 startMining() 中计算 speedMultiplier 时：
import net.minecraftforge.common.ForgeHooks;

// 替换当前的 getToolSpeed() + 手动计算
// 临时装备工具到玩家手上，调用 ForgeHooks.blockStrength()
float breakProgress = ForgeHooks.blockStrength(block, player, world, x, y, z);
mine.speedMultiplier = breakProgress * 30.0F; // 还原为乘数，因为 BASE_PROGRESS_PER_TICK = 1/30
```

但这有个问题：`ForgeHooks.blockStrength()` 会读取玩家当前手持物品和附魔效果，需要临时装备工具。

#### 方案 B：手动实现等价逻辑（更可控）

```java
// 在 startMining() 中：
private static float computeSpeedMultiplier(EntityPlayerMP player, ItemStack tool, 
        Block block, World world, int x, int y, int z) {
    int meta = world.getBlockMetadata(x, y, z);
    float hardness = block.getBlockHardness(world, x, y, z);
    
    if (hardness < 0) return 0; // 不可破坏
    
    // 1. 基础挖掘速度
    float breakSpeed;
    if (tool != null && tool.getItem() != null) {
        breakSpeed = tool.getItem().getDigSpeed(tool, block, meta);
        if (breakSpeed <= 1.0F) breakSpeed = 1.0F;
    } else {
        breakSpeed = 1.0F; // 空手
    }
    
    // 2. 效率附魔加成
    if (breakSpeed > 1.0F && tool != null) {
        int efficiencyLevel = EnchantmentHelper.getEfficiencyModifier(player);
        if (efficiencyLevel > 0) {
            breakSpeed += efficiencyLevel * efficiencyLevel + 1;
        }
    }
    
    // 3. 急迫药水效果
    if (player.isPotionActive(Potion.digSpeed.id)) {
        int amplifier = player.getActivePotionEffect(Potion.digSpeed.id).getAmplifier();
        breakSpeed *= 1.0F + (amplifier + 1) * 0.2F;
    }
    
    // 4. 挖掘疲劳效果
    if (player.isPotionActive(Potion.digSlowdown.id)) {
        int amplifier = player.getActivePotionEffect(Potion.digSlowdown.id).getAmplifier();
        breakSpeed *= 1.0F - (amplifier + 1) * 0.2F;
    }
    
    // 5. 远程挖掘：强制 onGround=true（不在地面惩罚 /5）
    //    远程挖掘时玩家实际可能悬空，但应该按在地面计算
    
    // 6. 远程挖掘：取消水下惩罚（/5）
    //    远程挖掘时玩家实际不在水下
    
    // 7. 能否采集判断
    boolean canHarvest = ForgeHooks.canHarvestBlock(block, player, meta);
    
    // 8. 最终速度
    float result;
    if (canHarvest) {
        result = breakSpeed / hardness / 30.0F;
    } else {
        result = breakSpeed / hardness / 100.0F; // 不能采集时慢 3.33 倍
    }
    
    return Math.max(result, 0.0F);
}
```

然后在 `startMining()` 中：
```java
mine.speedMultiplier = computeSpeedMultiplier(player, actualTool, block, world, x, y, z);
```

同时移除 `BASE_PROGRESS_PER_TICK` 的固定值，改为：
```java
// 每 tick 进度直接就是 speedMultiplier（已经是 breakSpeed/hardness/30 或 /100 的结果）
float increment = mine.speedMultiplier;
```

或者保留 `BASE_PROGRESS_PER_TICK = 1.0F`，让 `speedMultiplier` 直接作为每 tick 进度。

#### 关键改动点：

1. **移除 `hardness` 的 clamp**：`Math.max(hardness, 0.3F)` → 直接使用原值
2. **增加"不能采集"惩罚**：不能采集时 `/100` 而非 `/30`
3. **增加效率附魔加成**：`EnchantmentHelper.getEfficiencyModifier(player)`
4. **增加药水效果**：Haste / Mining Fatigue
5. **强制 onGround=true**：远程挖掘时模拟在地面
6. **取消水下惩罚**：远程挖掘时不在水下

### 2.2 使用原版方块破坏进度纹理

**目标文件**: `AnimationRenderer.java`, `InteractionViewModel.java`

**方案**: 将自定义 GL_LINES 渲染替换为调用 `World.destroyBlockInWorldPartially()`。

#### 客户端改动 (`InteractionViewModel.java`):

```java
// 当收到 S2CRtsMineProgressMessage(stage=0~9) 时：
public void applyMineProgress(int x, int y, int z, int stage) {
    if (stage < 0) {
        // 清除裂纹
        mc.theWorld.destroyBlockInWorldPartially(
            RTS_MINE_RENDER_ID, mineProgressX, mineProgressY, mineProgressZ, -1);
        mineProgressX = mineProgressY = mineProgressZ = 0;
        mineProgressStage = -1;
        return;
    }
    
    // 位置变化时清除旧裂纹
    if (mineProgressStage >= 0 && 
        (mineProgressX != x || mineProgressY != y || mineProgressZ != z)) {
        mc.theWorld.destroyBlockInWorldPartially(
            RTS_MINE_RENDER_ID, mineProgressX, mineProgressY, mineProgressZ, -1);
    }
    
    // ★ 调用原版方法，使用自定义实体 ID 避免与真实玩家冲突
    mc.theWorld.destroyBlockInWorldPartially(
        RTS_MINE_RENDER_ID, x, y, z, Math.min(9, stage));
    
    mineProgressX = x;
    mineProgressY = y;
    mineProgressZ = z;
    mineProgressStage = Math.min(9, stage);
}
```

其中 `RTS_MINE_RENDER_ID = 0x525453`（与原版 RTSbuilding 一致，ASCII "RTS"）。

#### 移除自定义渲染 (`AnimationRenderer.java`):

删除或注释掉 `renderMineProgress()` 方法中的自定义 GL_LINES 渲染代码。

#### 需要注意的问题：

1.7.10 中 `World.destroyBlockInWorldPartially()` 会通知所有 `IWorldAccess`（包括 `RenderGlobal`），而 `RenderGlobal.destroyBlockPartially()` 会将 `DestroyBlockProgress` 存入 `damagedBlocks` Map，然后在 `drawBlockDamageTexture()` 中渲染标准裂纹纹理。

**但有一个问题**：`destroyBlockInWorldPartially` 需要从服务端广播给客户端（通过 S2C packet），或者直接在客户端调用。1.7.10 中这个方法是 `World` 级别的，客户端可以直接调用（因为客户端有自己的 `WorldClient` 实例）。

实际上，1.7.10 中 `PlayerControllerMP.onPlayerDamageBlock()` 就是直接在客户端调用 `this.mc.theWorld.destroyBlockInWorldPartially()`，所以客户端调用是完全可行的。

### 2.3 增加首次点击延迟（可选）

**目标文件**: `RtsMineManager.java`

```java
// 在 ActiveMine 中增加字段：
int blockHitDelay; // 首次点击延迟

// 在 startMining() 中：
mine.blockHitDelay = 5; // 5 tick 延迟

// 在 onServerTick() 中：
if (mine.blockHitDelay > 0) {
    mine.blockHitDelay--;
    return; // 延迟期间不推进进度
}
```

**注意**：原版 RTSbuilding 1.21.1 没有这个延迟（因为是远程挖掘），所以这一步是可选的。如果要与原版 RTSbuilding 对齐，则不加延迟。

## 三、实施优先级

| 优先级 | 改动 | 影响 | 复杂度 |
|--------|------|------|--------|
| **P0** | 修复速度计算（不能采集惩罚、效率附魔、药水效果） | 修复"破坏过快"的核心问题 | 中 |
| **P1** | 使用原版裂纹纹理替换自定义 GL_LINES | 修复"没有原版破坏进度"的视觉问题 | 低 |
| **P2** | 移除 hardness clamp | 修复软方块速度异常 | 低 |
| **P3** | 增加首次点击延迟 | 与原版 MC 体验对齐（可选） | 低 |
